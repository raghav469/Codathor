What is the difference between == and === in JavaScript? Easy

== (loose equality) performs type coercion before comparison, while === (strict equality) compares both value and type without conversion. Examples: 5 == "5" // true (string converted to number) 5 === "5" // false (different types) null == undefined // true (special case) null === undefined // false (different types) 0 == false // true (false converted to 0) 0 === false // false (different types) Best Practice: Always use === for predictable, bug-free comparisons.

Explain closures in JavaScript with a practical example Medium

A closure is a function that has access to variables from its outer (enclosing) scope even after the outer function returns. function createCounter() { let count = 0; return function() { count++; return count; }; } const counter1 = createCounter(); const counter2 = createCounter(); console.log(counter1()); // 1 console.log(counter1()); // 2 console.log(counter2()); // 1 (separate closure) Practical Use Cases: - Module pattern for data privacy - Function factories - Event handlers that need access to outer variables - Callback functions with persistent state

What is event delegation and why is it useful? Medium

Event delegation is attaching a single event listener to a parent element instead of multiple listeners to child elements, using event bubbling. // Instead of this: document.querySelectorAll('.button').forEach(btn => { btn.addEventListener('click', handleClick); }); // Use this: document.getElementById('container').addEventListener('click', function(e) { if (e.target.matches('.button')) { handleClick(e); } }); Benefits: 1. Better performance (fewer event listeners) 2. Works with dynamically added elements 3. Less memory usage 4. Easier to manage and maintain Perfect for: Dynamic content, large lists, menu systems

Explain the JavaScript event loop Hard

The event loop is JavaScript's concurrency model that handles asynchronous operations in a single-threaded environment. Components: 1. Call Stack: Executes synchronous code 2. Web APIs: Handle async operations (setTimeout, DOM events, HTTP requests) 3. Callback Queue: Holds completed async callbacks 4. Event Loop: Moves callbacks from queue to stack when stack is empty Execution Order: console.log('1'); setTimeout(() => console.log('2'), 0); Promise.resolve().then(() => console.log('3')); console.log('4'); Output: 1, 4, 3, 2 Why? - Synchronous code runs first (1, 4) - Promises use microtask queue (higher priority) (3) - setTimeout uses macrotask queue (lower priority) (2) Key Points: - Microtasks (Promises) have higher priority than macrotasks (setTimeout) - Event loop only processes callbacks when call stack is empty

What are Promises and how do they work? Medium

Promises represent the eventual completion (or failure) of asynchronous operations. States: 1. Pending: Initial state 2. Fulfilled: Operation completed successfully 3. Rejected: Operation failed Creating a Promise: function fetchData() { return new Promise((resolve, reject) => { setTimeout(() => { const success = Math.random() > 0.5; if (success) { resolve('Data received'); } else { reject('Network error'); } }, 1000); }); } Using Promises: fetchData() .then(data => console.log(data)) .catch(error => console.error(error)) .finally(() => console.log('Request completed')); Promise Methods: - Promise.all(): Wait for all promises - Promise.race(): Return first settled promise - Promise.allSettled(): Wait for all, get all results - Promise.resolve()/reject(): Create resolved/rejected promises

Explain Python's GIL (Global Interpreter Lock) Hard

The GIL is a mutex that protects access to Python objects, preventing multiple threads from executing Python bytecode simultaneously. Key Points: 1. Only one thread can execute Python code at a time per process 2. Designed to simplify memory management (reference counting) 3. Impacts CPU-bound multi-threaded programs significantly 4. I/O-bound operations can release the GIL Impact on Performance: import threading import time def cpu_bound_task(): count = 0 for i in range(50000000): count += i # Sequential execution start = time.time() cpu_bound_task() cpu_bound_task() print(f"Sequential: {time.time() - start}") # Threading (won't improve CPU-bound tasks) start = time.time() t1 = threading.Thread(target=cpu_bound_task) t2 = threading.Thread(target=cpu_bound_task) t1.start() t2.start() t1.join() t2.join() print(f"Threading: {time.time() - start}") Workarounds: - Use multiprocessing for CPU-bound tasks - Use asyncio for I/O-bound tasks - Use C extensions that release the GIL - Consider alternative Python implementations (Jython, IronPython)

Explain list comprehensions vs generator expressions Medium

Both provide concise ways to create sequences, but with different memory and performance characteristics. Syntax Comparison: # List comprehension squares_list = [x**2 for x in range(10)] # Generator expression squares_gen = (x**2 for x in range(10)) Key Differences: 1. Memory Usage: import sys list_squares = [x**2 for x in range(1000)] gen_squares = (x**2 for x in range(1000)) print(sys.getsizeof(list_squares)) # ~9000 bytes print(sys.getsizeof(gen_squares)) # ~112 bytes 2. Evaluation: # List: Eager evaluation (all values computed immediately) numbers_list = [x for x in range(5)] print(numbers_list) # [0, 1, 2, 3, 4] # Generator: Lazy evaluation (values computed on demand) numbers_gen = (x for x in range(5)) print(numbers_gen) # print(list(numbers_gen)) # [0, 1, 2, 3, 4] 3. Reusability: # Lists can be iterated multiple times my_list = [x for x in range(3)] print(sum(my_list)) # 3 print(sum(my_list)) # 3 # Generators are consumed after one iteration my_gen = (x for x in range(3)) print(sum(my_gen)) # 3 print(sum(my_gen)) # 0 (generator exhausted) When to Use: - List comprehensions: Small datasets, need random access, multiple iterations - Generator expressions: Large datasets, memory constraints, single iteration

Explain Java's memory management and garbage collection Hard

Java's automatic memory management system handles object allocation and deallocation. Memory Areas: 1. Heap: Objects and instance variables 2. Stack: Method calls and local variables 3. Method Area: Class metadata, static variables 4. PC Register: Current executing instruction 5. Native Method Stack: Native method calls Generational Garbage Collection: 1. Young Generation: - Eden Space: New objects created here - Survivor Spaces (S0, S1): Objects that survived one GC cycle 2. Old Generation: - Long-lived objects promoted from young generation 3. Permanent Generation/Metaspace: - Class metadata (removed in Java 8) GC Process: 1. Minor GC: Cleans young generation - Fast and frequent - Most objects die young 2. Major GC: Cleans old generation - Slower, less frequent 3. Full GC: Cleans entire heap - Slowest, should be rare GC Algorithms: - Serial GC: Single-threaded, small applications - Parallel GC: Multi-threaded, throughput-focused - G1 GC: Low-latency, large heaps - ZGC: Ultra-low latency JVM Tuning: -Xms512m # Initial heap size -Xmx2g # Maximum heap size -XX:+UseG1GC # Enable G1 collector -XX:MaxGCPauseMillis=200 # Target pause time Memory Leaks in Java: - Static collections holding references - Unclosed resources (files, connections) - Inner classes holding outer references - ThreadLocal variables not cleared

What are Java Streams and how do they work? Medium

Java Streams (Java 8+) provide a functional approach to processing collections of data. Key Concepts: 1. Stream Pipeline: Source ‚Üí Intermediate Operations ‚Üí Terminal Operation 2. Lazy Evaluation: Operations are not executed until terminal operation 3. Immutable: Streams don't modify the original data source Example: List names = Arrays.asList("John", "Jane", "Jack", "Jill"); List result = names.stream() .filter(name -> name.startsWith("J")) // Intermediate .map(String::toUpperCase) // Intermediate .sorted() // Intermediate .collect(Collectors.toList()); // Terminal Intermediate Operations (lazy): - filter(): Filter elements based on predicate - map(): Transform elements - flatMap(): Flatten nested structures - distinct(): Remove duplicates - sorted(): Sort elements - limit(): Limit number of elements - skip(): Skip first n elements Terminal Operations (trigger execution): - collect(): Collect to collection - forEach(): Perform action on each element - reduce(): Reduce to single value - count(): Count elements - anyMatch()/allMatch()/noneMatch(): Boolean tests - findFirst()/findAny(): Find elements Advanced Example: Map> peopleByCity = people.stream() .filter(person -> person.getAge() > 18) .collect(Collectors.groupingBy(Person::getCity)); Benefits: - Functional programming style - Parallel processing capability - Readable and concise code - Lazy evaluation for performance

Explain the concept of Java interfaces and abstract classes Medium

Both provide abstraction in Java but serve different purposes. What is the Event Loop in Node.js? Hard

The Event Loop is the core of Node.js's non-blocking I/O model, enabling JavaScript to perform asynchronous operations. Event Loop Phases: 1. Timer Phase: Executes setTimeout() and setInterval() callbacks 2. Pending Callbacks: Executes I/O callbacks deferred to next loop iteration 3. Idle, Prepare: Internal use only 4. Poll Phase: Fetch new I/O events; execute I/O related callbacks 5. Check Phase: Execute setImmediate() callbacks 6. Close Callbacks: Execute close event callbacks Event Loop Example: console.log('Start'); setTimeout(() => console.log('Timer 1'), 0); setImmediate(() => console.log('Immediate 1')); process.nextTick(() => console.log('Next Tick 1')); Promise.resolve().then(() => console.log('Promise 1')); setTimeout(() => console.log('Timer 2'), 0); setImmediate(() => console.log('Immediate 2')); process.nextTick(() => console.log('Next Tick 2')); Promise.resolve().then(() => console.log('Promise 2')); console.log('End'); Output Order: Start End Next Tick 1 Next Tick 2 Promise 1 Promise 2 Timer 1 Timer 2 Immediate 1 Immediate 2 Priority Order: 1. process.nextTick() callbacks (highest priority) 2. Promise callbacks (microtasks) 3. Timer callbacks (macrotasks) 4. setImmediate() callbacks 5. I/O callbacks 6. Close callbacks Key Concepts: - Single-threaded event loop - Non-blocking I/O operations - Callbacks are queued and processed in order - Heavy CPU tasks can block the event loop - Use worker threads for CPU-intensive operations
